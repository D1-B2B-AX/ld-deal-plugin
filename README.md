# [Guide] 딜 판단 플러그인

## 🔹**1. 플러그인 목적: 문제 정의(페인&솔루션)**

- **페인(Pain)**: 매일 여러 딜이 진행 중이지만 슬랙·메일·구글시트·세일즈맵에 **딜 관련 데이터가 흩어져 있어 가치있는 정보로 바뀌지 못함**. 즉 결과로 "어떤 딜에 시간 더 써야 하나"의 판단 로직이 LD마다 직관에 머물러 정리되지 않음.
- **솔루션(Solution)**: **흩어진 딜 데이터를 가치있는 정보 (우선순위 점수·티어·핵심 시그널)로 변환.** 
/딜판단 한 번 실행하면 T1 집중·T2 관리·T3 지켜보기 3 영역 MD 리포트 생성. **LD가 스스로 판단 로직을 정리**할 수 있게 시각·구조적으로 도움.

## 🔹**2. 작동 원리 개요**

### 5개 데이터 소스 통합:

- **세일즈맵 본부장 DB (메인/출발점)** — 매일 새벽 3시 KST 자동 갱신. 
어떤 딜이 우선순위 판단 대상인지 결정하는 출발점. 진행 중인 딜만 골라내고(LOST·수주 완료 제외), 예상 체결액·수주 예정일·LD 본인이 매긴 성사 가능성·딜당 메모 본문을 가져옴.
- **슬랙 (메인/raw)** — b2b_2팀_견적제안·b2b_2팀_제안노트 채널. 
최근 2주 활동 (점수 영역)과 2026년부터 영업 사이클 전체 흐름 (거래 의지 LLM 판단 영역) 두 영역으로 나눠 봄. 회사명,딜명 검색 시 thread 전체를 자동으로 읽어 LD reply까지 누락 없이 회수. 
(견적서·제안서가 슬랙으로 전달된 사실(파일명·시각)도 시그널로 잡음.)
- **구글 드라이브 내 기획/견적 시트 (보조)** — 견적시트의 최종 탭에서 net·총액·갱신일을 가져와 딜 진행 활성도와 예상 체결액 검증에 활용. 기획문서는 LD별 정보 풍부도 및 활용 여부가 달라 코드 분기는 살리되 LD 케이스에 따라 활용도 조정 가능.
- **메일 (보조)** — 고객과의 양방향 응답 여부와 요구사항·일정 약속 추출. 
14일 윈도우(관찰 범위) 안에 고객이 회신하고 있는지, 메일 본문에 키워드("전달 예정"·"납기" "확정 예정" 등)가 박혀있는지 판단.
- **캘린더 (보조)** — LD 본인 캘린더의 딜 관련 미팅. 14일 윈도우. 
회사명/딜명 매칭 미팅과 소통 내용이 있으면 주간 액션 영역으로 표시.

## 🔹5. 딜판단 7가지 기준은 어떻게 도출됐나 (설계 흐름)

플러그인의 7기준은 세 영역에서 도출됨 — 표준 프레임워크 리서치 + 2팀 파트장 현실 피드백 + Won/Lost 역산.

**1) 표준 프레임워크 리서치 (B2B 기업 교육 영업 딜 우선순위 딥리써치 진)**

글로벌 B2B 영업의 표준 자격검증 프레임워크 3종을 먼저 리서치: BANT(예산·권한·필요·시기 — 단기 거래형), CHAMP(도전·권한·자금·우선순위 — 컨설팅형), MEDDPICC(지표·경제적 구매자·의사결정 기준·프로세스·서류·페인·챔피언·경쟁 — 엔터프라이즈형). 한국 시장 특수성 (법정 의무교육·분기 예산주기)·글로벌 사례 (Cornerstone·LinkedIn·Coursera 스코어링) 반영.

**2) 교육 2팀 의견 현실 피드백 (노을,민선님 회신)**

표준 프레임워크가 우리 영역 (B2B 기업교육)에 적용 가능한지 판단 및 교육 2팀 의견으로 현실 영역 짚음:

- 액티브 딜 분기 기준 — Proposal 준비 단계부터를 액티브 딜로 봐야 한다는 의견. 1·2차 f-up은 case by case로 양방향 변동이 잦아 분기 기준으로 부적합. Proposal 준비가 모든 케이스가 수렴하는 기점.
- 메인 기준(딜 금액·거래 의지) 설정 — 현실적으로 딜판단에 우선적으로 고려되는 기준 메인으로 설정.

***이 두 영역이 현재 플러그인의 액티브 게이트 + 메인/서브 위계 (1번·6번 메인, 나머지 서브)로 박혀 있습니다.***

**3) Won/Lost 딜 키워드 설정** 

실제 수주·실주 딜의 슬랙 메시지에서, 두 결과를 가르는 변별력 있는 키워드를 도출했습니다. 
이 키워드를 7가지 기준 중 일부(주로 6번 거래 의지)에 반영해 스코어링에 활용합니다.

## 🔹6. 7가지 기준 상세 — 각 기준이 어떤 영업 본질을 잡나

🔸**1번. 딜 금액** ⭐**(메인, 30점)**

딜 금액은 가장 단순하면서도 가장 신뢰도 높은 우선순위 시그널. 
세일즈맵 ‘예상 체결액’ 컬럼에 LD가 직접 입력 → 신뢰도 100%. 어떤 LLM 추론도 이 정보를 능가하지 X. 
그래서 메인 위계 중 하나로 박힘 (6번 거래 의지와 동등 30점). 
☑️점수: raw → 정보 변환: 원→억 환산 후 5단계 구간 (3억+ 30점 / 1억+ 20점 / 5천만+ 15점 / 3천만+ 10점 / 그외 5점). 견적시트 net·총액으로 검증·협상 시그널 보강.

🔸**2번. 파이프라인 단계 (서브, 18점)**

파이프라인 단계는 딜이 지금 어느 진척도에 있는지 봄. 세일즈맵에 enum으로 박힘 (Proposal 준비·송부·2차 f-up·보완 Proposal·최종 f-up·매출 집계 예정). 
☑️점수: 매출 집계 예정 18점 / 최종 f-up 16점 / 보완 Proposal 14점 / 2차 f-up 13점 / Proposal 송부 10점 / Proposal 준비 5점.

🔸**3번. 마감 임박도 (서브, 22/-10점)**

마감 임박도는 수주 예정일까지 얼마나 남았는지 봄. 가까울수록 LD가  신경을 써야 하는 비중이 달라짐. 
세일즈맵에 *제안서 마감일*과 *수주 예정일* 두 영역이 박혀있는데 채움 비율이 다름. 
3단계 트리로 *가장 정확한 시점*을 골라옴: 제안서 마감일 → 수주 예정일 → LLM이 슬랙·메일 raw에서 추출. ☑️점수: D-7 22점 / D-14 18점 / D-30 11점 / D-60 7점 / 60+ 4점 / 기한초과 -10점.

🔸**4번. 고객 가치 (서브, 23점)**

고객 가치는 이 거래가 단발에 그칠지·확장 가능성이 큰 지를 봄. 
☑️점수-4 차원 분해: 거래 이력 15점 (기고객·재의뢰는 가산) + 핵심기업 매칭 2점 (주거래 대기업 및 계열사에 소량 가산) + 레퍼런스 가치 3점 (후속 교육·다른 사업장 확장 시그널) + 확장 가능성 3점 (같은 고객사 별건 추가 패턴). 

🔸**5번. 소통 활성도 (서브, 27/-27점)**

소통 활성도는 지금 이 딜이 진행 중인지·정체 중인지 봄. 
☑️점수-4 축 종합: 슬랙 thread 14일 활동량 15점 (30건+ 15점 / 15+ 12점 / 5+ 8점 / 1+ 4점) + 기획,견적시트 갱신 12점 (7일 이내 12점 / 14일 9점 / 30일 6점) + 메일 양방향 응답 8점 (고객이 답하면 가산) + 메모 풍부도 5점 (3건+ 5점 / 1~2건 2점). 
메인 두 축(슬랙·시트) 모두 침묵 시 보조 축(메일·메모)으로 보조 점수 max 13점. 메인 한 축 침묵 시 -5점 페널티 (메일 양방향 활발 시 +5점 상쇄).

🔸**6번. 거래 의지 ⭐ (메인, 30점)**

거래 의지는 이 거래가 진짜 성사될지의 진실 영역. 1번 딜 금액과 동등 메인 (결정론 데이터와 LLM 추론이 동등함 — 사용자 통찰). 
실제 Won&Lost 딜 케이스의 슬랙 데이터 raw 역산으로 *진짜 변별력 키워드* 도출. 수주에 흔한 표면 키워드는 자동 제외 — Won vs Lost 차이만 시그널화.
(LLM이 슬랙·메일·메모 raw에서 시그널 카테고리 판정.)
[수주 시그널] W1 결정 직전·W2 활발한 협의·W3 의도 강도·W4 재의뢰·확장·W5 의사결정 구조. 
[실주 시그널] L1 권한 약함·L2 도메인 비매칭·L3 예산 한계·L4 확장 부재·L5 활동량 부재. 
☑️점수: C1 결정 시그널 12점 + C2 권한·예산 10점 + C3 활동·관심 8점 합산. 

🔸**7번. 딜 출발점 (서브, 20점)**

딜 출발점은 어떻게 우리에게 들어왔는지 봄. 출발점에 따라 진척 가능성이 다름
☑️점수: RFP·소개·기고객 재의뢰 20점 (명확한 의지·신뢰 영역) / 인바운드 10점 (일반적 진행) / 아웃바운드·다수 비교 5점 (진척 보장 X) / 가격 비교만 0점 (거래 가능성 낮음).

## 🔹7. 스코어링 — 점수 룩릅·티어·grounding

**7가지 기준 점수가 모이면 만점 170 = 메인 60 + 서브 110**.

- 메인 60점 — 1번 딜 금액(30) + 6번 거래 의지(30)
- 서브 110점 — 2번 단계(18) + 3번 마감(22) + 4번 고객 가치(23) + 5번 소통(27) + 7번 출발점(20)

▪️**티어 구분**

- **T1** ≥ 87 — 오늘 즉시 대응
- **T2** ≥ 55 — 이번 주 관계 육성
- **T3** < 55 — 주기적 모니터링 (정체·지켜보기)

▪️**액티브 게이트 (스코어링 대상 결정)**

두 조건을 모두 만족하는 딜만 점수를 매깁니다:

2. 성사 가능성 ∉ LOST · 확정

1차 f-up·SQL은 진척이 모호하고, LOST·확정(Won)은 수주 후 운영 영역이라 제외됩니다. 게이트를 통과하지 못한 딜은 별도 추적 트랙으로 빠집니다.

▪️**grounding 비교 (LD 직관 vs LLM 분석)**

LD가 세일즈맵에 직접 매긴 성사 가능성과 LLM이 슬랙·메모를 보고 판단한 거래 의지를 비교합니다. 
LD 진실이 우선이고, 분석이 LD 직관과 명백히 어긋날 때만 검토 권장 메시지가 박힙니다.

- LD 확정·높음 + LLM high·mid → match
- LD 확정·높음 + LLM low → mismatch (검토 권장)
- LD 낮음 + LLM mid·low → match
- LD 낮음 + LLM high → mismatch (검토 권장)

한 단계 차이는 노이즈로 흡수하고, 두 단계 차이만 진짜 갭 시그널로 봅니다.

## 🔹9. 커스터마이징

플러그인은 특정 LD의 환경에 맞춰 박혀 있지만, **모두 본인 환경으로 수정 가능합니다.** 
**핵심은 settings.json — 다른 LD는 본인의 채널·라벨·캘린더로 바꾸면 그대로 작동합니다. 
모든 수정은 Claude Code 대화창에 자연어로 시키면 즉시 반영됩니다.**

**(1) settings.json — 본인 환경 수정 (메인)**

지금 박힌 특정 LD 설정을 본인 영역으로 바꿉니다:

- target_ld(이름·회사 이메일·팀·슬랙 user ID·캘린더 ID)
- gmail_labels — 본인 메일 라벨
- slack_channels — 본인 영업 채널
- drive_folders — 본인 견적·기획문서 폴더

자연어 예시: "내 슬랙 채널에 #b2b_3팀_견적제안 추가해줘" / "내 회사 이메일·슬랙 user ID 박아줘"

**(2) 점수·임계값·prompt 톤도 자연어로 수정 가능**

- 점수 룰: "T1 임계값을 87에서 90으로 올려줘"
- prompt 톤: "T1 헤드라인 영역 더 짧게 해줘"
- 일회성 필터: "오늘 결과만 ○○사 빼고 보여줘"
- 본질 룰: "거래 의지 LLM mid를 LD 낮음으로만 매핑해줘"

**(3) 본인 깃 fork·공유 (선택)**

본인 수정 영역을 보존하거나 다른 LD와 공유 원할 때만:

- 본인 깃에 보존: "내 GitHub에 fork해서 올려줘" → Claude Code가 본인 fork에 push
- 다른 LD와 공유: 본인 fork URL 슬랙 공유
- AX팀 main 반영: PR 보내거나 슬랙으로 요청

**(4) v2 영역 (현재 미적용, 향후 개발 필요하다고 고려되면 LD가 시도 가능)**

지금은 안 박혀 있지만 추후 추가 가능한 기능:

- 자연어 피드백 layer — LD 자연어 피드백 → 다음 회차 점수에 반영 (4단계 + 5 조작 유형)
- 슬랙 첨부 본문 추출 — 엑셀 파싱·PDF 텍스트 (현재는 첨부 메타만 활용)
- 세일즈맵 메일 이력 활용 — 본부장 DB API 연동 또는 dump 추가 요청
- OM→LD DM 정보 회수 — 별도 채널 추출 도구
