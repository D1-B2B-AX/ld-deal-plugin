# 🎼 딜 판단 플러그인 — 오케스트레이터

> LD(영업 리더)가 매일 **SQL 단계 딜 우선순위를 빠르게 파악**할 수 있도록, 6개 서브 스킬을 지휘하여 MD 리포트를 생성하는 메인 스킬.
> 본 플러그인은 **운영(Won) 제외, 세일즈 우선순위 판단에 집중**한 경량 버전입니다.

## 핵심 원칙

1. **"검증은 코드, 판단은 사람"** — 점수 계산·검증은 스크립트, 근거 생성·액션 추천은 LLM
2. **SQL 전용 집중** — Won 딜은 건드리지 않음 (풀버전 `deal-priority-plugin`이 처리)
3. **LD 다중 사용자 지원** — `settings.json`으로 본인 정보·채널·임계값 관리
4. **단일 진입점, 완결된 산출물** — 한 번 실행 → MD 리포트 하나

## 실행 조건

사용자가 `/딜판단` 실행하면 아래 절차를 **순서대로** 수행합니다.
초기 대화나 범위 선택 없음 — SQL 전용이라 항상 같은 방식.

**권장:** `claude --dangerously-skip-permissions`로 시작하면 중간 승인 없이 자동 실행.

## STEP 0. 자동 알림 메커니즘 (4/30 신규 — 윤지님 anchor 우선주의 정합)

> **목적**: 빌더가 GitHub에 새 버전 push 후, 사용자가 *모르는 사이* 룰이 바뀌는 것 방지. 사용자 인지·결정 가시화.

### 동작 흐름

```
1. /딜판단 실행 직후 — Claude가 자동 체크
   cd ~/.claude/commands/ld-deal-plugin
   git fetch origin --quiet
   LOCAL=$(git rev-parse HEAD)
   REMOTE=$(git rev-parse origin/main)

2. 분기:
   ├─ LOCAL == REMOTE → "최신 버전입니다 ✓" → PHASE 0 진행
   │
   └─ LOCAL != REMOTE → 사용자에게:
       a. 변경 요약 표시:
          git log --oneline HEAD..origin/main
          # 예: "fix: 결손 매트릭스 ④ 임계값 조정"
       b. "업데이트 하시겠습니까? (y/n)"
       c. 사용자 응답:
          ├─ y → git pull → "업데이트 완료" → PHASE 0 진행
          └─ n → "현재 버전으로 진행 (구버전)" → PHASE 0 진행
```

### Claude 실행 instruction (요약)

매 `/딜판단` 진입 시:

1. **현재 디렉토리 확인** — `~/.claude/commands/ld-deal-plugin` 인지
2. **fetch + 비교** — Bash 도구로 위 명령 실행
3. **변경 감지 시 사용자 안내** — 변경 커밋 메시지 + 변경 파일 리스트 표시
4. **사용자 응답 받음** — "업데이트 진행" 또는 "현재 버전 유지" 답변 대기
5. **응답 따라 분기 실행** — pull 또는 그대로 진행

### 처리 원칙 (윤지님 anchor 우선주의 정합)

- **사용자 인지 가시화** — 변경이 있다는 사실을 *사용자가 명시 인지* 후 진행
- **사용자 결정 가시화** — 업데이트 여부를 *사용자가 명시 결정*. 자동 적용 X
- **변경 요약 명시** — 무엇이 바뀌었는지 커밋 메시지 + 파일 리스트로 보여줌
- **거부 시 graceful** — "n" 응답 시 *현재 버전으로 정상 진행*. 강제 차단 X

### 안전 처리

- git 폴더가 아닌 경우 (예: 빌더 로컬 작업판) → STEP 0 skip + 로그 1줄
- 네트워크 실패 (`git fetch` 오류) → "업데이트 체크 실패, 현재 버전 진행" + 로그
- LOCAL이 REMOTE보다 *앞선* 경우 (빌더 로컬 미푸시) → "현재 버전이 최신" 표시 + 진행

### 작동 시점

- ⏸ **현재 (4/30 빌더 작업)**: 코드만 박음 — GitHub 미배포라 작동 X
- ✅ **5/7 GitHub 배포 후**: 사용자 install·`git clone` 환경에서 실제 작동
- ✅ **운영 단계**: 매 `/딜판단` 실행 시 자동 작동

## PHASE 2. 보조 데이터 수집 (병렬 4개)

스킬 1의 deals[]를 인풋으로, **4개 스킬 동시 실행.**

### 스킬 2: 캘린더 조회
- `settings.target_ld.calendar_id` 기준 이번 주 + 다음 주 이벤트 1회 조회
- 고객사명 매칭으로 딜별 분류

### 스킬 3: 슬랙 조회
- **팀 자동 매핑**: `settings.target_ld.team`("1팀"/"2팀")에 따라 `slack_common_channels + slack_team_channels[team]` 자동 조합
- OR 통합 검색으로 2~3회 호출 (속도 최적화)
- 2팀 LD = 공통 4 + 2팀 5 = 9개 채널

### 스킬 4: 지메일 조회
- `settings.data_sources.gmail_label` 내에서 고객 이메일로 양방향 검색
- 응답 상태 판별 (주고받는 중 / 미응답 / 교환 없음)

### 스킬 5: 드라이브 기획시트 조회
- `settings.target_ld.email` 소유 기획/싱크업 문서 검색
- **60행까지 읽기 (정보 풍부화)**
- **8개 항목 파싱**: 배경·마감·견적·일정·요구·경쟁·의사결정·리스크

**에러 처리:** 개별 스킬 실패해도 나머지 계속 진행. 전체 실패해도 스킬 1 데이터만으로 PHASE 3 가능.

## PHASE 2.5. 보조 소스 기반 데이터 보완 (스크립트 + LLM 두 단계)

> **스킵 금지.** 세일즈맵이 출발점이지만, 스킬 2~5에서 더 최신/정확한 정보 확인 시 보완.
> **5/6 갱신**: 단일 LLM 단계였던 것을 *결정론 자동 머지(2.5a)* + *LLM 보완(2.5b)* 두 단계로 분리.

### PHASE 2.5a — 자동 머지 [스크립트] (5/6 신규)

```bash
python scripts/enrich_external.py \
  --phase2 runtime/phase2_active_deals.json \
  --skill2 runtime/skill2_calendar.json \
  --skill3 runtime/skill3_slack.json \
  --skill4 runtime/skill4_gmail.json \
  --skill5 runtime/skill5_drive.json \
  -o runtime/phase2_enriched.json
```

처리:
- `deal_id` 기준 left join — 스킬 결과 없는 딜은 phase2 원본 필드만 유지
- 스킬 결과에 phase2에 없는 deal_id는 무시 (warn 로그)
- 매칭 카운트 stderr 출력

머지 대상 필드 (스크립트에 박힘):
- 스킬 2 (캘린더): `calendar_results`
- 스킬 3 (슬랙): `slack_results`·`slack_summary`·`activity_flag`·`slack_thread_count_14d`·`slack_attachments` ← 5/6 첨부 메타 신규
- 스킬 4 (지메일): `email_results`·`email_summary`·`customer_responded`·`last_email_days`
- 스킬 5 (드라이브): `drive_results`·`last_quote_sheet_updated_days`

### PHASE 2.5b — LLM 보완 (필수)

자동 머지 후 *결정론으로 잡히지 않는 영역*만 LLM이 판단.

### 보완 체크리스트

**1. 소통 상태 (`last_touch_days`, `customer_responded`, `email_status`)**
- 세일즈맵 메모 날짜보다 메일 교환이 더 최근이면 → 메일 기준으로 갱신
- 메일에서 고객 회신 확인되면 → `customer_responded = true`
- 슬랙에서 활발한 논의 → 소통 활성으로 판단

**2. deal_origin (LLM 판단)**
- 반드시 세일즈맵 메모 + 메일 기반으로 판단
- 허용 값 7개: RFP / 소개·추천 / 기고객 재의뢰 / 인바운드 / 아웃바운드 / 다수 업체 비교 / 가격 비교만
- 근거 명확한 경우만 상위 카테고리, 애매하면 "인바운드"(기본)

**3. win_probability (세일즈맵 직접 값)**
- 세일즈맵 `성사 가능성` 필드의 "높음"/"낮음" 그대로 추출
- 스킬 1에서 이미 전달. LLM 재판단 X (LD 본인 판단 우선)

**4. 주간 액션 (`has_action_this_week`)**
- 캘린더 미팅 자동 반영
- 슬랙/메일에서 "이번 주 내 회신" 등 액션 확인 시 true

## PHASE 3. 스코어링 (직렬 3단계)

### 3a. 데이터 병합 [스크립트]
```bash
python scripts/merge_deals.py \
  --skill1 skill1_raw.json \
  --skill2 skill2_calendar.json \
  --skill3 skill3_slack.json \
  --skill4 skill4_email.json \
  --skill5 skill5_drive.json \
  --today YYYY-MM-DD \
  --settings config/settings.json \
  -o deals.json
```

- 5개 중간파일 → `deals.json` 단일 스키마로 정규화
- SQL 전용 필터 (방어적)
- `strategic_keywords` settings에서 로드 → `is_strategic_customer` 계산

### 3b. 스코어링 [스크립트]
```bash
python scripts/calculate_score.py deals.json --settings config/settings.json > scored_deals.json
```

**7개 기준 (만점 170, 4/30 갱신):**
| 기준 | 배점 | 비고 |
|---|---|---|
| 1. 딜 금액 | +30 | `예상 체결액` (4/30) + `Net(%)` 보조 |
| 2. 파이프라인 단계 | +18 | (5/10/13/14/16/18) — 4/29 노을님 결정 |
| 3. 마감 임박도 | +22 | **3단계 트리** (제안서 마감일 → 수주 예정일 → LLM 추출) |
| 4. 고객 가치 | +23 | 4차원 (거래15 + 규모2 + 레퍼3 + 확장3) |
| 5. 소통 활성도 | +27/-27 | 두 축 결합 (슬랙 thread + 견적서 갱신) |
| 6. 거래 의지 | +30 | LLM 카테고리 (C1·C2·C3) + **`성사 가능성` grounding 비교** |
| 7. 딜 출발점 | +20 | (win_probability boost는 6번 grounding 영역으로 분리) |

**티어 분류 (4/30 갱신):** T1 ≥ 87 / T2 ≥ 55 / T3 < 55

**⏸ T1 후보 뱃지 폐기 (4/30, §2-11)** — 이중 가산 위험으로 폐기. 단순 총점 기반 티어만.

**액티브 게이트 (4/30 신규):** 단계 ∈ 액티브 6종 AND `성사 가능성 ≠ LOST` (LOST 79% 차단)

### 3c. 검증 [스크립트]
```bash
python scripts/verify_scores.py scored_deals.json
```

**6항목 검증 (4/30 갱신)** — 누락 / 합산 / 범위 / **만점 170 일치** / 티어 87/55 / 중복. 실패 시 최대 2회 재시도.

**3종 변화:**
- `added` — 신규 SQL 딜
- `removed` — SQL에서 사라짐 (DB 조회로 Won/Lost/Convert 구분)
- `stage_changed` — 파이프라인 단계 변경

**SQL→Won 구분:** removed 각 항목의 `current_status`를 세일즈맵 DB에서 조회 → 🎉 수주 성공 별도 표시

### 4c. MD 리포트 생성 [스크립트]
```bash
python scripts/generate_md.py \
  --scored scored_deals.json \
  --changes changes.json \
  --settings config/settings.json \
  --today YYYY-MM-DD \
  --runtime <초단위>
```

**출력:** `outputs/summary_report_YYYYMMDD.md`

**자동 정리:** retention_days(14일) 초과 리포트 자동 삭제
