# 스킬 5: 드라이브 기획시트·견적서 조회 (SQL 단계 전용)

> 딜별 **기획시트** 또는 **교육 견적서** 를 Google Drive에서 찾아 핵심 정보를 파싱하는 보조 스킬.
> **수주 전 딜(SQL) 우선순위 판단에 가장 풍부한 정보원** — 세일즈맵 메모보다 더 많은 맥락이 여기에 담긴다.

## 역할

스킬 1(세일즈맵)의 **보조 스킬**이지만 실질적으로 **수주 전 딜의 가장 풍부한 정보원**.
- 기획시트: 딜 배경·요구사항·경쟁·의사결정·리스크
- 견적서: 최신 견적 금액·조건·교육 일정

## 대상

**SQL 단계 딜만 실행.** 본 플러그인은 Won 이후 운영 관련 문서를 다루지 않음.
**"싱크업 문서"는 검색에서 제외** (Won 딜 운영 관련 성격).

## 인풋

| 파라미터 | 출처 | 설명 |
|---------|------|------|
| `organization_name` | 스킬 1 아웃풋 | 고객사명 (Drive 검색 키워드) |
| `deal_name` | 스킬 1 아웃풋 | 과정명 매칭용 |
| `owner_email` | `settings.target_ld.email` | 담당자 이메일 (소유자 필터) |

## 처리 흐름

```
STEP 1: 소유자 기준 Drive 검색 (1회) — "기획" OR "견적서"
  ↓
STEP 2: 매칭 안 된 딜만 fallback 검색 (고객사명)
  ↓
STEP 3: 딜-파일 매칭 (고객사명 + LLM 보조 판단)
  ↓
STEP 4: 파일 타입 판정 (기획시트 / 견적서)
  ↓
STEP 5: 타입별 탭 선별 + 읽기
  ↓
STEP 6: LLM 파싱 — 타입별 차등 항목 추출
```

## STEP 1~2: Drive 검색

### 1차 검색 (소유자 기준, 1회)
```
검색: (이름에 "기획" OR "견적서") + 소유자 = LD 이메일 + 스프레드시트
```
**"싱크업"은 의도적으로 제외** — Won 딜 관련 문서는 본 플러그인 범위 밖.

### 2차 검색 — fallback (매칭 안 된 딜만)
```
검색: 이름에 "{고객사명}" + ("기획" OR "견적서") + 스프레드시트
소유자 필터 없음 (OM 등 다른 사람이 만든 파일 포함)
```

### 검색 키워드 (최종)
| 키워드 | 커버하는 파일명 |
|--------|--------------|
| `기획` | 기획시트, 기획문서, 기획 문서 |
| `견적서` | 교육견적서, 견적서 |

~~`싱크업`~~ — **제거됨** (Won 운영 관련)

## STEP 3: 딜-파일 매칭 (불일치 대응 강화)

세일즈맵 딜명 ↔ 드라이브 파일명이 **다를 수 있음.**

**실제 케이스 예시:**
- 세일즈맵: `현대카드_PM/서비스 기획자 대상 교육`
- 드라이브: `패스트캠퍼스_교육견적서_현대카드_프로덕트1팀_Figma 교육`
- 공통: 고객사명 "현대카드"만 일치, 과정 상세 다름

**매칭 로직:**

```
1순위: 고객사명(organization_name) 매칭
  → 1건 매칭 → 자동 연결

2순위: 고객사 1건 × 파일 여러 개
  → LLM이 딜명 토큰 vs 파일명 토큰 유사도 비교
  → 가장 비슷한 파일 선택
  → 예: "PM/서비스 기획자" vs "Figma 교육" 같은 경우도 LLM이 맥락 판단

3순위: 동일 고객사 같은 딜에 기획시트 + 견적서 둘 다 존재
  → 기획시트 = 메인 (풍부 파싱)
  → 견적서 = 보조 (최신 견적 금액만 추가)
  → 둘 다 planning_sheet 필드에 반영

4순위: 매칭 실패
  → "⚠️ 세일즈맵 딜명과 드라이브 파일명 불일치" 안내
  → planning_sheet: null로 진행 (나머지 스코어링은 정상)
```

## STEP 4: 파일 타입 판정

```
파일명에 "견적서" 포함 → 타입 = "견적서"
파일명에 "기획" 포함    → 타입 = "기획시트"
```

## STEP 5: 타입별 탭 선별 + 읽기

### 기획시트: 탭 화이트리스트·블랙리스트 방식

**읽기 (상단 40행):**
| 탭 이름 키워드 | 가치 |
|---|---|
| RFP / 제안요청 / 요구사항 | 🔴 핵심 (마감일·예산·과업) |
| 사업 / 배경 / 현황 | 🔴 핵심 (고객 맥락·니즈) |
| 기획 / 아이데이션 / 설계 | 🟠 상단 (배경·목표) |
| 견적 (YYMMDD 패턴) | 🟠 필요 (금액) — **최신 1개만** |

**스킵 (읽지 않음):**
- index / 계산기 / 면세·과세 / 강사료 기준
- 히스토리 / 과거 / 템플릿 / 참고자료
- 시간별 모듈 상세 (커리큘럼 풀 테이블)

**범위:** 각 탭 **상단 40행**만 (기존 60행에서 축소 — 토큰·시간 절감)

### 견적서: 최신 "견적" 탭 1개만

**읽기 (전체 탭):**
- 탭 이름에 "견적(YYMMDD)" 또는 "견적_날짜" 패턴만 매칭
- 여러 개면 **가장 최근 날짜만**
- 구조 단순해 전체 탭 읽어도 부담 작음

**스킵 (공통 양식):**
- index / 면세·과세 기준 정책 / 강사료 산정 기준 / 계산기
- 모두 공통 양식이라 딜별 특이 정보 없음

**효과:** 견적서 파일당 탭 읽기 5~9회 → **1회**로 대폭 단축

## STEP 6: LLM 파싱 — 타입별 차등 항목

### 기획시트 (8개 항목, 풍부)

| 파싱 항목 | 설명 |
|---------|------|
| `deal_context` | 딜 배경 2~3줄 |
| `proposal_deadline` | RFP/제안서 마감 (절대 날짜) |
| `quote_amount` | 견적 규모 + 조건 |
| `education_schedule` | 확정/미확정/추후 협의 |
| `key_requirements` | 고객이 원하는 것 (리스트) |
| `competitor_situation` | 경쟁사·차별화 |
| `decision_process` | 누가/언제/어떻게 결정 |
| `risk_signals` | 딜 성사 위협 요인 |

**파싱 프롬프트 지시사항:**
- "커리큘럼 모듈별 상세 내용(M1-1, M2-3 등 시간표)은 제외"
- "배경·목표·일정·규모·견적·경쟁만 추출"
- "시간표·실습 예시 무시"

### 견적서 (3~4개 항목, 간단)

| 파싱 항목 | 설명 |
|---------|------|
| `quote_amount` | 최신 견적 금액·조건 (최신 탭 기준) |
| `education_schedule` | 교육 일정 (있으면) |
| `deal_context` | 고객·과정명 간단 요약 |
| 나머지 | `null` (견적서에 없는 정상) |

### 같은 딜에 둘 다 있을 때 (옵션 C)

- 기획시트를 메인으로 8개 항목 풍부 파싱
- 견적서에서 **`quote_amount`만** 별도 추출 → 기획시트 견적보다 더 최신이면 업데이트
- `planning_sheet.source` 필드에 "기획시트+견적서 병합" 표기

## MCP 호출

```
파일 검색: mcp__workspace-mcp__search_drive_files
시트 정보: mcp__workspace-mcp__get_spreadsheet_info
탭 읽기:   mcp__workspace-mcp__read_sheet_values (range "A1:Z40")
```

## 속도 예상 (최적화 후)

| 단계 | 호출 수 | 비고 |
|------|--------|------|
| 1차 검색 (소유자) | 1회 | "기획" OR "견적서" |
| 2차 fallback | 0~3회 | 매칭 안 된 딜만 |
| 시트 정보 조회 | 매칭된 파일 수 (약 3~5회) | |
| 탭 읽기 — 기획시트 | 파일당 2~4회 (화이트리스트 탭만) | |
| 탭 읽기 — 견적서 | 파일당 1회 (최신 견적만) | |
| **총 예상** | **약 10~20회** | 기존 20~40회 대비 50% 감소 |

## 알려진 제약·주의사항

| 항목 | 내용 |
|------|------|
| SQL 단계만 | Won 딜 문서는 조회 스킵 |
| 싱크업 제외 | Won 운영 관련 문서 배제 (본 플러그인 범위 외) |
| 탭 40행 제한 | 40행 이후 데이터는 포기 (토큰·시간 우선) |
| 탭 구조 비정형 | 사람마다 탭명·구조 다름 → 화이트리스트 키워드로 커버 |
| 소유자 불일치 | 다른 사람이 만든 문서는 fallback 검색으로 커버 |
| 딜명 불일치 | 고객사명 매칭 + LLM 보조 판단 |

## LD 체크 포인트

1. 기획시트·견적서 네이밍 규칙 일관성 있는지 (매칭 정확도에 직접 영향)
2. 세일즈맵 딜명과 드라이브 파일명이 주기적으로 싱크 맞는지
3. 견적서 내 "견적(YYMMDD)" 탭 네이밍 패턴 유지 (날짜 포함)

## 기존 플러그인과의 차이

| 항목 | `deal-priority-plugin` (풀버전) | **`ld-deal-plugin` (본)** |
|---|---|---|
| 검색 키워드 | "기획" OR "싱크업" | **"기획" OR "견적서"** (싱크업 제외) |
| 대상 | SQL + Won 기획시트·싱크업 | **SQL만, 기획시트·견적서** |
| 탭 처리 | 전체 탭 읽기 (30행) | **화이트리스트 탭만 (40행)** |
| 견적서 처리 | 미분리 | **별도 타입, 최신 탭 1개만** |
| 파싱 항목 | 5개 | **기획시트 8개 / 견적서 3~4개 차등** |
| 딜명 불일치 | 단순 매칭 | **LLM 보조 매칭 + 경고** |
| 둘 다 있을 때 | 기획시트만 | **기획시트 메인 + 견적서 최신가 병합** |
| 호출 횟수 | 20~40회 | **10~20회** (약 50% 감소) |
