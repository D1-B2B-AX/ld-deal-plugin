## 0-2. 4/30 정보원 매핑 ((target_ld) 실데이터 검증 후 갱신)

> **갱신 사유**: 4/30 본부장 DB schema 확인에서 *4/29 anchor에 박힌 일부 정보원이 0% 채움*으로 무효화. 실측 채움 비율 기반으로 정보원 정정.

### 핵심 정보원 (실측 기반)

| 점수 | 1차 정보원 | 채움 (진짜 7건) | 백업·보조 |
|---|---|---|---|
| 1번 딜 금액 | **`예상 체결액`** (deal 컬럼) | 100% | `Net(%)` (있을 때 가산) |
| 3번 마감 임박도 | **3단계 트리** (아래) | — | — |
| 6번 거래 의지 | LLM (슬랙 + memo) | — | **`성사 가능성` grounding** 100% |

### 3번 마감 임박도 트리 (4/30 사용자 통찰)

1. **`제안서 마감일`** (deal 컬럼) — 가장 정확. 현재 0% 채움 → 향후 LD 입력 시 자동 활용
2. **`수주 예정일`** (deal 컬럼) — 백업. 100% 채움 / 단 초기 추정·변동 미반영 가능
3. **슬랙·메일 LLM 추출** — 실시간 보강. 마감 키워드 ("제출 마감"·"전달 마감"·"전달 예정"·"납기"). 6번 LLM 호출에 통합 출력으로 포함 (토큰 효율).

### 6번 거래 의지 grounding (4/30 신규 발견 / 5/6 매핑 정밀화)

- LD가 직접 매기는 `성사 가능성` (JSON array `["확정"|"높음"|"낮음"|"LOST"]`) — 액티브 7건 100% 채움
- LLM 결과(high/mid/low)와 *비교 검증* — 일치도 측정 → 산출물 노출 → 불일치 케이스가 곧 우선순위 검토 영역
- LLM 비결정성 위험을 *추가 grounding 데이터*로 완화 (운영일지 §8 한계 영역의 핵심 진전)

**매핑 매트릭스 (5/6 옵션 C — 비대칭 1단계 흡수)**:

| LD 입력 | LLM high | LLM mid | LLM low |
|---|---|---|---|
| 확정 | match | match | mismatch (검토 권장) |
| 높음 | match | **match** ⭐ | mismatch (검토 권장) |
| 낮음 | mismatch (검토 권장) | **match** ⭐ | match |
| LOST | (액티브 게이트 차단) | | |

⭐ = 4/30 N/A 처리되던 케이스(지멘스·디어포스 mid)가 5/6부터 정상 match — LD 진실 우선 룰 정합.

**룰**: 한 단계 차이는 노이즈로 흡수, 두 단계 차이만 시그널. mid는 애매한 영역이라 LD enum 모두 흡수. 박힌 위치 = `scripts/calculate_score.py:check_grounding_match()`.

### 텍스트 raw 메인·보조 (4/30 정정)

- **메인**: 슬랙 (4/29 결정 그대로)
- **보조** (4/30 신규): **`memo` 테이블** (dealId 매칭, 딜당 평균 3.3개 × 평균 418자, max 1567자) — 6번 LLM raw 입력 결합

### 폐기·약화 (4/30 검증)

- ❌ **폐기**: `기획시트 링크` (deal 컬럼, 0% 채움) / `금액` (deal 컬럼, 6%만 채움) / `마감일` (의미 모호 + 1/7 채움)
- ⏸ **v2 영역** (현재 미활용, 향후 입력 시 자동): `제안서 마감일`
- ⚠️ **약화**: 드라이브 기획문서 검색 — `memo` 테이블이 보조 메인 흡수, (target_ld) 케이스 기획문서 적음

### 5/6 갱신 — 슬랙 raw 수집 룰 (lead 배분 패턴 차단)

- **룰**: 검색에서 hit한 메시지가 thread의 일부면 thread 전체(root + 모든 reply)를 raw에 포함
- **사유**: lead 배분 thread root는 *팀장·파트장이 작성*. LD reply에 회사명·@LD 멘션 *둘 다 없는 경우 잦음* → 회사명 단독 키워드 검색은 reply 누락
- **검증 사례**: 4/30 지멘스 검증에서 4/21 (target_ld) reply 2건(커리큘럼·강사 작업) 누락 → 5/6 본 룰로 차단
- **구현**: skill 3 SKILL.md STEP 2.5 신설 (`slack_read_thread` MCP 도구 사용)

## 12. 결과물 표시 형태 (4/29 결정)

### 12-1. 큰 그림 — b + a 결합 (분류 + 행동)

페인·솔루션 정합:
- 페인: "딜 데이터를 가치있는 정보로 바꾸지 못해 *판단 로직이 정리되지 않은 것*"
- 솔루션: "딜 데이터를 가치있는 정보로 바꿔 *스스로 판단 로직을 정리*할 수 있게 돕는 것"
- → 분류(판단) + 행동(액션) 결합으로 **판단 → 액션 흐름** 제공

### 12-2. 결과물 구조

**[상단부 — 분류 (전 티어)]**
- T1 / T2 / T3 / 비액티브 — 전 티어 딜 목록
- T2·T3는 **짧은 한 줄 + 분류** (예: "디어포스 — 거래의지 mid / 마감 D-30")
- 사유: LD가 *왜 그 티어인지* 빠르게 파악 → Play 4 학습 효과

**[하단부 — T1 풀 상세만]**
- 순서: **A 왜 T1? → B 추천 다음 액션 → C 점수 상세 표**

### 12-3. T1 딜 풀 상세 형태 (시안)

````
## [딜명] [T1]

### 왜 T1? (정보로 변환 결과)

**🎯 결정 시그널 — high/mid/low**
└ 근거: [데이터 → 정보 변환 한 줄]

**👥 권한·예산 — high/mid/low**
└ 근거: [데이터 → 정보 변환 한 줄]

**📊 활동·관심 — high/mid/low**
└ 근거: [데이터 → 정보 변환 한 줄]

**💰 기타 강한 시그널**
└ [마감·금액·소통 활성도 등 6번 외 강한 시그널]

### 추천 다음 액션
- [액션 리스트]
````

| 결정 | 정신 |
