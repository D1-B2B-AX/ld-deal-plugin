**버전**: v0 (2026-04-29)

## 0. 본 문서의 의도 (metalevel 원칙)

본 문서의 자문·빨간 깃발·갈래·룰은 **"강제 통과 룰"이 아니라 "생각할 시점을 제공하는 환기 도구"**입니다.

- 시스템(본 문서)은 **멈춤·환기만**
- 최종 판단은 **빌더(전정현)**
- 잘못된 방향 시그널이 보일 때 **잠시 멈춰 본질을 자각**한 후 결정
- 적절한 피드백·질문이 보이면 그걸 토대로 빌더가 결정

엄격한 통과 강요는 개발 진척을 저해하고, 외부 변동 요인·실제 데이터 변수를 흡수하지 못하게 합니다 (운영일지 §8 한계 경험). **anchor는 *큰 방향 결정*에만 작동**하고, 세부 기능은 데이터 보고 유연하게 결정.

## Play 1. Anchor 박기

### 1.1. 페인 (4/22 본부장 컨펌)

> "딜 데이터를 가치있는 정보로 바꾸지 못해 판단 로직이 정리되지 않은 것"

### 1.2. 솔루션 (4/22 본부장 컨펌)

> "딜 데이터를 가치있는 정보로 바꿔, 스스로 판단 로직을 정리할 수 있게 돕는 것"

### 1.3. 7기준 (4/29 옵션 B 확정)

| # | 기준 | 위계 |
|---|---|---|
| 1 | 딜 금액 | **메인** |
| 2 | 파이프라인 단계 | 서브 |
| 3 | 마감 임박도 | 서브 |
| 4 | 고객 가치 | 서브 |
| 5 | 소통 활성도 | 서브 |
| 6 | 거래 의지 | **메인** |
| 7 | 딜 출발점 | 서브 |

### 1.4. 위계 (4/30 갱신 — (target_ld) 실데이터 검증 후)

**스코어링 위계 (변경 없음 — 4/29 결정 유지)**
- 메인: 1번 딜 금액 · 6번 거래 의지
- 서브: 2·3·4·5·7번

**데이터 위계 (4/30 정정 — (target_ld) 액티브 7건 schema 측정 기반)**
- **출발점**: 세일즈맵 (본부장 대시보드 DB — 매일 새벽 갱신, 수동 다운로드)
- **텍스트 raw 메인**: 슬랙 (`#b2b_2팀_견적제안` · `#b2b_2팀_제안노트` 등)
- **텍스트 raw 보조** (4/30 신규 발견): **`memo` 테이블** (dealId 매칭 — 딜당 평균 3.3개 × 평균 418자, max 1567자)
- **숫자·날짜 시그널**: 견적서 (드라이브 `1.[Sales]-02.견적` 폴더 — 텍스트 X, 최종 탭 net·총액·갱신 날짜만)
- **보조**: 메일 · 캘린더

**4/30 검증 결과 폐기·약화:**
- ❌ **폐기**: `기획시트 링크` (세일즈맵 컬럼) — (target_ld) 액티브 7건 중 0% 채움
- ⏸ **v2 영역 (현재 활용 X, 향후 LD 입력 시 자동 활용)**: `제안서 마감일` (현재 0% 채움)
- ⚠️ **약화**: 드라이브 기획문서 검색 — `memo` 테이블이 보조 메인 흡수, (target_ld) 케이스 기획문서 자체 적음. 코드 분기는 살리되 1차 활용 폐기
- ❌ **폐기**: `마감일` 컬럼 — 의미 모호 + 1/7 채움 + 다른 컬럼들과 다른 값

### 1.5. 분기 룰 (4/30 EOD 갱신)

- **액티브 딜만 스코어링 대상**
  - 액티브 정의 (4/30 EOD 갱신 — 두 조건 AND):
    1. **단계 ∈** {Proposal 준비, Proposal 송부, 2차 f-up, 보완 Proposal, 최종 f-up, 매출 집계 예정}
    2. **AND `성사 가능성` ∉ {LOST, 확정}** ⭐ 4/30 EOD (Won·Lost 모두 차단)
  - 사유 (4/29 그대로): 1·2차 f-up은 딜별·LD별 case by case / 단계 양방향 변경 빈번 (역행) / Proposal 준비 = 모든 케이스 수렴 기점
  - 사유 (4/30 신규): (target_ld)이 LOST 결과를 별도 단계로 이동시키지 않고 *성사 가능성 표시만* 하는 패턴. 단계 매칭만으론 LOST 다수 포함됨 (실측: 33건 단계 액티브 중 26건이 LOST = 79%).
  - 사유 (4/30 EOD 추가 — 검증 시점): `성사 가능성 = "확정"`은 closed/Won 처리. *수주 후 운영 단계*는 우선순위 판단 영역 X. (target_ld) 검증 시 답지에서 사용자가 "확정 = won 처리로 보임"으로 명시 검증 제외.
  - LD 자연어 피드백으로 액티브 승격 가능
- **비액티브 딜은 별도 추적 트랙** (스코어링 X — 1차 f-up · SQL · LOST · 확정/Won)

### 1.6. 핵심 정보원 (4/30 신규 — 실측 채움 비율 기반)

| 점수 | 1차 정보원 | 진짜 7건 채움 | 백업·보조 |
|---|---|---|---|
| 1번 딜 금액 | `예상 체결액` (deal 컬럼) | **100%** | `Net(%)` (있을 때 가산) |
| 3번 마감 임박도 | 3단계 트리 (아래) | — | — |
| 6번 거래 의지 | LLM (슬랙 + memo) | — | **`성사 가능성` grounding 비교** |

**3번 마감 임박도 트리 (4/30 사용자 통찰):**
1. **`제안서 마감일`** (deal 컬럼) — 가장 정확. *현재 0% 채움이지만 향후 LD 입력 시 자동 활용*
2. **`수주 예정일`** (deal 컬럼) — 백업. *현재 100% 채움. 단 초기 추정·변동 미반영 가능*
3. **슬랙·메일 LLM 추출** — 실시간 보강. 마감 키워드 ("제출 마감", "전달 마감", "전달 예정", "납기" 등). 6번 LLM 호출에 통합 출력으로 포함 (토큰 효율).

**6번 grounding 룰:**
- LD가 직접 매기는 `성사 가능성` (JSON array `["확정"|"높음"|"낮음"|"LOST"]`) — 액티브 7건 100% 채움
- 6번 LLM 결과(high/mid/low)와 *비교 검증* — 일치도 측정 → 산출물에 함께 노출 → 불일치 케이스가 곧 우선순위 검토 영역

## Play 2. Drift Prevention (회차마다 환기)

### 2.1. 자문 체크리스트 — reference 가이드 (의무 X)

큰 변경(새 변수·새 차원·새 기준) 도입 시 *점검용*으로 활용. 평상시엔 본인 판단.

1. **페인·솔루션 부합** — 이 변경이 페인('데이터→정보 변환 결손')을 어떻게 해결하는가? 솔루션('자립 판단 돕기')의 어느 요소에 기여하는가?
2. **위계 정합** — 새 변수·시그널이 메인(1번 금액·6번 거래 의지)인지 서브인지 명확한가? 서브가 메인 weight를 앞지르지 않는가?
3. **데이터 위계 정합** — 슬랙 raw가 의사결정 메인 데이터로 활용되는가? 세일즈맵은 출발점, 메일·시트·캘린더는 보조로만?
4. **분기 룰 정합** — 이 딜이 액티브 게이트를 통과한 케이스인가? 비액티브 딜에 스코어링 적용하려는 흐름은 없는가?
5. **페인 문장 literal** — 큰 패러다임 점프 없이 페인 문장이 가리키는 결함 위치 그대로 매핑하는가? (메모리 `feedback_pain_statement_literal`)

### 2.2. 빨간 깃발 — 등장 시 잠시 멈춤·재검토 (강한 가드)

```
🚩 ① "어차피 이 정도 차이는 괜찮아"           → anchor 침식 신호
🚩 ② "비액티브 딜에도 스코어링 한 번 적용"     → 분기 룰 위반
🚩 ③ "슬랙 비중 줄이고 메일 메인으로"          → 데이터 위계 위반
🚩 ④ "메인/서브 위계 폐기 평면화"              → 위계 결정 번복 위험
🚩 ⑤ "신규 8번·9번 기준 추가"                  → 7기준 확정 위반
```

### 2.3. 빨간 깃발 발동 시 처리 흐름

```
빨간 깃발 등장
  ↓
즉시 멈춤
  ↓
판단:
  (a) anchor 갱신해서 진행 (정당한 사유 + 본부장/사용자 합의)
  (b) 변경 폐기 (anchor 침식 위험)
  (c) 변경 변형 (anchor 부합하게 재설계)
```

## Play 3. Diff Decision Tree (외부 reference 비교 검증 시)

### 3.1. 원칙

외부 reference(LD 수동 판단·다른 빌더 결과물·노을·(target_ld) 인사이트 등)와 갭 발견 시 **"빌더가 틀렸다" 결론 X**. 아래 5갈래 중 어디 속하는지 분류 → 그에 맞는 처리.

### 3.2. 5갈래 (딜판단 특화)

#### 갈래 1. 본질 안 + 도구 처리 가능 → 도구 보강
- 시그널: "이 변환 룰·시그널이 명확히 누락됐다"
- 처리: `docs/information_criteria.md`에 변수·룰 추가
  - 결정론(스코어링) — 점수 weight·기준 보강
  - LLM 추론(6번 거래 의지) — 의도 강도 시그널 변수 추가
- 사례: 4/29 역산 결과로 6번 시그널 7개 → 10개+로 확장

#### 갈래 2. 본질 안 + 도구 처리 불가 → 자연어 피드백 layer 영역 (원칙 1) ⏸ v2 영역
- 시그널: "데이터로 담을 수 없는 무형정보·구두 결정·real-time 변동"
- 처리 (v2): 자연어 피드백으로 LD가 직접 조정 (시스템은 해석·반영만)
- 사례: "이 딜은 담당자 톤이 긍정적이라 가능성 높다" → LD 자연어로 +N boost
- v1 처리: 본 갈래 발견 시 *6번 grounding 비교*로 우선순위 검토 영역 표시 (자동 조정 X)

#### 갈래 3. 의도적 제외 → 정상 작동
- 시그널: "이 영역은 처음부터 다루지 않기로 했다"
- 처리: 갭 = 설계 의도. 보강 X
- 사례: 비액티브 딜에 우선순위 안 매김 / 메일 1차 스코어링 제외

#### 갈래 4. LD idiom 차이 → 표준 + 개인화 분리 (딜판단 특화 신설)
- 시그널: "다른 LD는 이렇게 본다"
- 처리: 표준 7기준은 유지 / 개인 `config/settings.json`에서 조정 가능 영역
  - 슬랙 채널 가중치, 키워드 가중치, 임계값 조정 등
- 출처: 윤지님 인사이트 #1 "AX팀이 표준 틀, 마무리는 LD가 본인 영역에 맞춰 커스터마이징"

#### 갈래 5. reference가 다른 종류 → 비교 frame 부적절
- 시그널: "이건 애초에 우리 도구와 다른 영역의 reference"
- 처리: 비교 자체 안 함
- 사례: 운영일지 결과물(누락 감지)을 딜판단(우선순위)과 직접 비교

### 3.3. 처리 흐름

```
외부 reference 갭 발견
  ↓
5갈래 중 분류
  ↓
갈래 1·2 → 도구·layer 보강
갈래 3·4·5 → 보강 X (정상·표준 유지·frame 부적절)
  ↓
어느 경우든 빌더 anchor는 흔들리지 않음
```

## Play 4. Baseline as Starting Point

### 4.1. 원칙

본 plugin이 가진 baseline(7기준 점수 룩룩·임계값·변수 카탈로그)은 **"정답" 아닌 "출발점"**. 사용자(LD)가 진화시킬 수 있는 메커니즘 설계가 본질.

> baseline 정확도(grounded 정도)보다 **진화 메커니즘 작동 여부가 더 중요한 검증 항목**.

### 4.2. baseline 구성

- 7기준 점수 룩룩 (`docs/information_criteria.md`에 명세)
- 18 변수 카탈로그 (결정론 11 + LLM 7) — 어제 4/28 §10.7
- baseline 출처: **4/29 역산 데이터** (Won 4 + Lost 6 슬랙 케이스, `docs/역산_원본_데이터_260429.md`)

### 4.3. 진화 메커니즘 (원칙 1 — 자연어 피드백 layer) ⏸ v2 영역 (2026-04-30 — MVP 제외)

> **MVP 진입 결정 (4/30)**: v1에서 자연어 피드백 layer 미구현. 6번 grounding (`성사 가능성` 비교)이 일부 역할 흡수. 본 섹션 설계는 v2 진입 시점에 그대로 활용.

- 클로드: 7기준 스코어링 + reason 제시
- LD: 자연어 피드백 입력 (예: "이 딜은 담당자 톤이 긍정적이라 +N boost")
- LLM: 자연어 해석 → 해당 딜 스코어/티어에 adjustment 반영
- 재계산 + "이 자연어가 이렇게 반영됐습니다" 근거 표시
- LD: 해석 확인 → 승인 or 수정

3 하위원칙:
1. **LLM 해석 투명성** — 블랙박스 금지
2. **양방향 확인 루프** — LD가 정정 가능
3. **기존 스코어링 보존 + Layer 방식** — 결정론 유지, 자연어는 위에 덧대는 adjustment layer

### 4.4. 검증 항목

- ❌ "baseline 점수가 LD 직관과 100% 일치하는가" (정확도 함정)
- ✅ "LD 자연어 피드백이 다음 회차 baseline에 자연스럽게 반영되는가" (진화 작동) ⏸ *v2 영역*
- ✅ "LD가 시스템이 자기 판단을 *돕는다*고 느끼는가" (페인 본질 해결) — v1은 *결과물 신뢰 + 6번 grounding* 비교로 평가

## Play 5. External Reference Trap Avoidance

### 5.1. 원칙

외부 reference는 **anchor 대체물이 아닌 비교 자료**. 풍부함 매칭 압력·새 frame 박힘 위험을 의식적으로 차단.

### 5.2. Reference 종류별 처리

#### 종류 A. 파트장·본부장 가이드 (윤지님·박원우님)
- 처리: 비교 자료 + 사용자 명시 결정 항목만 채택
- 미명시 항목은 `docs/확인_리스트.md`로 추적, 즉답 회피
- 채택 = anchor 갱신 전제 (Play 2 빨간 깃발 연계)
- 사례: 4/29 윤지님 회신 12개 항목 → 명시 결정 6개만 반영, 미명시 6개는 추적

#### 종류 B. (target_ld) LD 수동 판단 (테스트 대상)
- 처리: 본 plugin v1 검증 reference. Play 3 5갈래로 분류
- "LD 판단이 정답" X / "비교 자료" O
- 자연어 피드백 layer로 흡수 (Play 4 진화 메커니즘)

#### 종류 C. 다른 LD 판단 (다른 팀 LD)
- 처리: 표준 vs 개인화 분리 (Play 3 갈래 4)
- 표준 anchor 영향 X / `config/settings.json`에서만 조정

#### 종류 D. 다른 빌더 plugin·외부 도구
